# Security Policy

## Supported Versions

We release patches for security vulnerabilities in the following versions:

| Version  | Supported          |
| -------- | ------------------ |
| Latest   | :white_check_mark: |
| < Latest | :x:                |

## Reporting a Vulnerability

If you discover a security vulnerability within this project, please follow these steps:

1. **Do not** open a public issue for the vulnerability.
2. Email security details to the repository maintainer or use GitHub's private vulnerability reporting feature.
3. Include the following information:
   - Type of vulnerability
   - Full paths of source file(s) related to the vulnerability
   - Location of the affected source code (tag/branch/commit or direct URL)
   - Any special configuration required to reproduce the issue
   - Step-by-step instructions to reproduce the issue
   - Proof-of-concept or exploit code (if possible)
   - Impact of the issue, including how an attacker might exploit it

## Response Timeline

- You can expect an initial response within 48 hours.
- We will investigate and confirm the vulnerability.
- We will work on a fix and coordinate the release with you.
- Once the fix is released, we will publicly acknowledge your responsible disclosure (unless you prefer to remain anonymous).

## Security Best Practices

When using this software:

1. Keep all dependencies up to date
2. Follow secure coding practices
3. Use environment variables for sensitive information
4. Regularly review and update security configurations
5. Enable two-factor authentication where applicable

## Third-Party Dependencies

This project relies on various third-party packages. We regularly monitor and update these dependencies to address known vulnerabilities. Users should also keep their installations up to date.
